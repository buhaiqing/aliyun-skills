## ECS 常见故障模式

### Pattern: ECS-001 — CPU 突增型过载

| 属性 | 内容 |
|------|------|
| **触发指标** | CPUUtilization |
| **触发阈值** | >= 95% |
| **典型时间** | 业务高峰期、定时任务执行时段 |
| **关联指标** | LoadAverage 同步升高，memory_usedutilization 可能正常或略高 |
| **根因** | 1. 应用突发流量 2. 定时任务 3. 恶意请求/攻击 |
| **诊断步骤** | 1. 检查网络流量是否同步增加 2. 通过 ECS Cloud Assistant 查看 top 进程 3. 检查安全组是否有异常来源 IP |
| **修复方案** | 1. 临时升配 2. 增加 SLB 后端扩容 3. 优化慢查询或定时任务调度 |
| **预防措施** | 1. 配置 CPU 趋势告警 2. 配置弹性伸缩 3. 优化应用代码 |

### Pattern: ECS-002 — IO Wait 型卡顿

| 属性 | 内容 |
|------|------|
| **触发指标** | LoadAverage |
| **触发阈值** | > vCPU * 2 |
| **典型特征** | LoadAverage 高但 CPUUtilization < 50% |
| **关联指标** | DiskUsage 高，IOPSUsage 高 |
| **根因** | 1. 磁盘 IO 瓶颈 2. 磁盘空间不足导致 IO 效率下降 |
| **诊断步骤** | 1. 检查 DiskUsage 和 IOPSUsage 2. 通过 ECS 云助手执行 iostat 3. 检查是否有大文件写入或日志暴涨 |
| **修复方案** | 1. 清理磁盘空间 2. 升级磁盘类型 3. 优化应用 IO 模式 |
| **预防措施** | 1. 配置 DiskUsage 告警 2. 配置日志轮转 3. 使用 ESSD 云盘 |

### Pattern: ECS-003 — 内存泄漏

| 属性 | 内容 |
|------|------|
| **触发指标** | memory_usedutilization |
| **触发阈值** | >= 90% |
| **典型特征** | memory_usedutilization 持续单调上升，重启后恢复正常 |
| **关联指标** | CPUUtilization 可能正常或略高 |
| **根因** | 1. 应用内存泄漏 2. 缓存未设置过期时间 3. 大对象未释放 |
| **诊断步骤** | 1. 检查 memory_usedutilization 趋势（是否单调上升）2. 通过 Cloud Assistant 查看内存占用 top 进程 3. 检查应用日志是否有 OOM |
| **修复方案** | 1. 重启应用（临时）2. 修复内存泄漏代码 3. 增加内存或优化缓存策略 |
| **预防措施** | 1. 配置 memory_usedutilization 趋势告警 2. 配置 OOM 自动重启 3. 代码审查内存管理 |

## RDS 常见故障模式

### Pattern: RDS-001 — 连接数耗尽

| 属性 | 内容 |
|------|------|
| **触发指标** | ConnectionUsage |
| **触发阈值** | >= 90% |
| **典型特征** | ConnectionUsage 高但 CPUUsage 低 |
| **关联指标** | CPUUsage < 30%，QPS/TPS 可能正常或下降 |
| **根因** | 1. 应用连接池配置过大 2. 连接未正确关闭 3. 慢查询导致连接长时间占用 |
| **诊断步骤** | 1. 检查 ConnectionUsage vs CPUUsage 2. 委托 DAS 获取活跃连接和睡眠连接比例 3. 检查慢查询日志 |
| **修复方案** | 1. 重启数据库（临时）2. 优化应用连接池配置 3. 修复连接泄漏代码 4. 优化慢查询 |
| **预防措施** | 1. 配置 ConnectionUsage 告警 2. 配置连接池监控 3. 定期审查慢查询 |

### Pattern: RDS-002 — CPU 型性能瓶颈

| 属性 | 内容 |
|------|------|
| **触发指标** | CpuUsage |
| **触发阈值** | >= 80% |
| **典型特征** | CpuUsage 和 QPS/TPS 同步升高 |
| **关联指标** | IOPSUsage 可能同步升高 |
| **根因** | 1. 突发高并发查询 2. 复杂查询/全表扫描 3. 缺少索引 |
| **诊断步骤** | 1. 检查 QPS/TPS 趋势 2. 委托 DAS 获取 SQL 诊断报告 3. 检查慢查询日志 |
| **修复方案** | 1. 优化慢查询 SQL 2. 增加索引 3. 读写分离 4. 升配实例规格 |
| **预防措施** | 1. 配置 CpuUsage 趋势告警 2. 定期 DAS 巡检 3. SQL 上线前审核 |

### Pattern: RDS-003 — 磁盘空间不足

| 属性 | 内容 |
|------|------|
| **触发指标** | DiskUsage |
| **触发阈值** | >= 90% |
| **典型特征** | DiskUsage 持续上升，可能伴随 IOPSUsage 升高 |
| **根因** | 1. 数据量增长 2. Binlog 未清理 3. 临时表过大 4. 日志文件膨胀 |
| **诊断步骤** | 1. 检查 DiskUsage 趋势 2. 委托 DAS 空间分析 3. 检查 Binlog 保留策略 |
| **修复方案** | 1. 清理 Binlog 2. 删除无用数据/归档历史数据 3. 扩容磁盘 4. 优化大表 |
| **预防措施** | 1. 配置 DiskUsage 告警 2. 配置自动 Binlog 清理 3. 定期数据归档 |

## SLB 常见故障模式

### Pattern: SLB-001 — 后端服务器过载

| 属性 | 内容 |
|------|------|
| **触发指标** | DropConnection |
| **触发阈值** | > 0 |
| **典型特征** | DropConnection 增加，后端 ECS CPU/Memory 高 |
| **关联指标** | InstanceActiveConnection 可能达到上限 |
| **根因** | 1. 后端 ECS 资源不足 2. 后端 ECS 健康检查失败 3. 连接数超过后端处理能力 |
| **诊断步骤** | 1. 检查 SLB 后端服务器健康状态 2. 检查后端 ECS CPU/Memory 3. 检查 SLB 连接数限制配置 |
| **修复方案** | 1. 增加后端 ECS 实例 2. 升级后端 ECS 规格 3. 优化应用性能 4. 调整 SLB 权重 |
| **预防措施** | 1. 配置后端 ECS 资源告警 2. 配置弹性伸缩 3. 优化健康检查配置 |

### Pattern: SLB-002 — 网络丢包

| 属性 | 内容 |
|------|------|
| **触发指标** | DropPacketRX / DropPacketTX |
| **触发阈值** | > 0 |
| **典型特征** | 丢包增加但后端 ECS 正常 |
| **根因** | 1. 带宽达到上限 2. 安全组/ACL 限制 3. 网络抖动 |
| **诊断步骤** | 1. 检查 SLB 带宽配置 2. 检查安全组规则 3. 检查 VPC 网络配置 |
| **修复方案** | 1. 升级带宽 2. 优化安全组规则 3. 更换网络类型 |
| **预防措施** | 1. 配置带宽监控告警 2. 定期检查安全组规则 |

## Redis 常见故障模式

### Pattern: Redis-001 — 连接泄漏

| 属性 | 内容 |
|------|------|
| **触发指标** | ConnectionUsage |
| **触发阈值** | >= 90% |
| **典型特征** | ConnectionUsage 高但 QPS 低 |
| **根因** | 1. 应用连接未关闭 2. 连接池配置不合理 3. 短连接频繁创建 |
| **诊断步骤** | 1. 检查 ConnectionUsage vs QPS 2. 检查应用连接池配置 3. 检查是否有大量空闲连接 |
| **修复方案** | 1. 修复连接泄漏 2. 优化连接池配置（使用长连接）3. 重启 Redis（临时） |
| **预防措施** | 1. 配置连接数告警 2. 代码审查连接管理 3. 使用连接池监控 |

### Pattern: Redis-002 — 大 Key 问题

| 属性 | 内容 |
|------|------|
| **触发指标** | MemoryUsage |
| **触发阈值** | >= 90% |
| **典型特征** | MemoryUsage 高，某些操作延迟大 |
| **根因** | 1. 单个 Key 值过大 2. 集合类型元素过多 3. 缓存未设置过期 |
| **诊断步骤** | 1. 委托 DAS 缓存分析 2. 检查大 Key 分布 3. 检查 Key 的内存占用 |
| **修复方案** | 1. 拆分大 Key 2. 设置合理的过期时间 3. 优化数据结构 4. 升级实例规格 |
| **预防措施** | 1. 配置内存告警 2. 定期大 Key 扫描 3. 代码审查 Key 设计 |

## 跨产品级联故障模式

### Pattern: CASCADE-001 — 单点 ECS 过载引发级联故障

```
[Timeline]
T1: ECS i-001 CPUUtilization 飙升至 100%
T2: SLB lb-001 DropConnection 增加（后端 i-001 无响应）
T3: RDS rm-001 ConnectionUsage 增加（应用重试导致连接堆积）
T4: 用户感知服务不可用

[根因]
ECS i-001 单点过载 → SLB 健康检查失败 → 流量未切换 → 应用重试 → RDS 连接堆积

[诊断]
1. CMS 检查 i-001 多指标（CPU + Memory + Load）
2. SLB 检查后端健康状态
3. RDS 检查连接数和活跃查询
4. DAS 获取 i-001 和 rm-001 的诊断报告

[修复]
1. 临时：重启 i-001 或手动切换 SLB 权重
2. 长期：增加 SLB 后端实例 + 配置弹性伸缩 + 优化应用性能
```

### Pattern: CASCADE-002 — 数据库慢查询引发应用雪崩

```
[Timeline]
T1: RDS rm-001 CPUUsage 升高（慢查询）
T2: 应用响应时间增加
T3: 应用连接池耗尽
T4: ECS CPU 升高（线程阻塞）
T5: SLB DropConnection 增加

[根因]
RDS 慢查询 → 应用响应慢 → 连接池耗尽 → 线程阻塞 → 级联故障

[诊断]
1. DAS 获取慢查询报告
2. CMS 检查 RDS CPU + ConnectionUsage 趋势
3. 检查 ECS LoadAverage（线程阻塞特征）

[修复]
1. 临时：重启应用释放连接
2. 长期：优化慢查询 SQL + 增加索引 + 读写分离
```

## 知识库使用指南

### 如何应用知识库

1. **告警触发时**：根据 namespace + metric 查找对应 Pattern
2. **多指标关联时**：检查是否符合 Pattern 中的"关联指标"特征
3. **诊断时**：按照 Pattern 的"诊断步骤"执行
4. **修复时**：优先执行"修复方案"中的临时措施，再实施长期方案
5. **复盘时**：将实际故障与知识库 Pattern 对比，更新或新增 Pattern

### 知识库维护

| 动作 | 触发条件 | 负责人 |
|------|----------|--------|
| 新增 Pattern | 发现新的重复故障模式 | Agent / 运维团队 |
| 更新 Pattern | 根因或修复方案有变化 | Agent / 运维团队 |
| 淘汰 Pattern | 架构变更导致 Pattern 不再适用 | 运维团队 |
| 验证 Pattern | 每季度回顾诊断准确率 | 运维团队 |
