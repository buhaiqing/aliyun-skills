# Troubleshooting — 架构评审故障排查

> 覆盖 12+ 常见错误场景，按数据源和报告阶段组织。

---

## 1. 快速诊断流程

```
报告无法生成
    │
    ├── 数据源问题?
    │      ├── topo-discovery 不可用 → 降级为用户描述
    │      ├── advisor-ops 不可用 → 跳过该数据源
    │      ├── cms-ops 无数据 → 跳过指标评估
    │      └── billing-ops 权限不足 → 跳过成本量化
    │
    ├── 用户输入不足?
    │      ├── 信息不足以识别模式 → 追问
    │      └── 约束条件不明确 → 追问
    │
    └── 报告逻辑错误?
           ├── 架构模式分类错误 → 重新采集数据
           ├── WAF 评分偏差大 → 检查评分规则
           └── 方案不可执行 → 重新设计
```

## 2. 数据源错误

### 2.1 topo-discovery 不可用

**症状**: 委托 `alicloud-topo-discovery` 失败或返回空数据

**可能原因**:
1. `topo-discovery` Skill 未部署或不可用
2. RAM 权限不足，无法执行资源发现
3. 账号下无资源或资源未被发现
4. 网络不通

**诊断步骤**:
```
1. 检查 topo-discovery 是否可用:
   检查是否有该 Skill 并尝试委托一次简单查询

2. 检查 RAM 权限:
   确认 AK 是否具有资源发现所需的 RAM 策略
   (ReadOnlyAccess 或具体产品 Describe 权限)

3. 检查网络:
   确认 Agent 运行环境能访问阿里云 API
```

**恢复动作**:
1. 降级为用户自行描述架构 topology
2. 在报告中标注 "架构拓扑基于用户描述，未经实际数据验证"
3. 设置 confidence 为 `low`

### 2.2 advisor-ops 不可用

**症状**: 获取 Advisor 巡检结果失败

**可能原因**:
1. `aliyun-cli-advisor` 插件未安装
2. RAM 权限不足 (`advisor:Describe*`)
3. Advisor 服务暂时不可用

**诊断步骤**:
```
1. 检查插件: 运行 `aliyun advisor version`
2. 检查权限: 尝试 `aliyun advisor GetProductList`
3. 检查服务: 检查 status.aliyun.com
```

**恢复动作**:
1. 跳过 advisor-ops 相关的检查项
2. 基于 topo-discovery 的配置信息做定性评估
3. 在报告中标注 "Advisor 数据不可用，WAF 评分基于资源配置的定性分析"

### 2.3 cms-ops 无数据

**症状**: 委托 `alicloud-cms-ops` 获取监控指标返回空

**可能原因**:
1. 云监控未开通（首次使用需要开通）
2. 被查资源的监控数据未被采集
3. 资源已停止/释放，无最新监控数据
4. RAM 权限不足

**诊断步骤**:
```
1. 检查云监控状态: 运行 `aliyun cms DescribeMetricMetaList`
2. 检查特定资源: 尝试查询其他指标确认
3. 检查权限: 确认有 cms:Describe* 权限
```

**恢复动作**:
1. 跳过 Performance 支柱的量化指标评估
2. 基于资源规格（规格型号、网络带宽等配置）做定性分析
3. 在报告中标注 "Performance 评估未包含实际监控数据"

### 2.4 billing-ops 权限不足

**症状**: 获取成本数据失败

**可能原因**:
1. RAM 策略未授权 `billing:Describe*` 操作
2. 使用了子账号且主账号未开通费用中心授权

**诊断步骤**:
```
1. 检查 RAM 权限:
   确认是否有 billing 相关策略
2. 检查账号类型:
   子账号需要主账号授权查看费用信息
```

**恢复动作**:
1. 跳过成本量化分析
2. 基于资源规格估算大概成本
3. 在报告中标注 "成本分析未包含实际账单数据"

## 3. 用户输入问题

### 3.1 描述过于模糊

**症状**: 用户只说 "帮我看一下我的系统" 但没有系统详情

**诊断**: 用户输入中缺乏架构分析所需的关键信息

**恢复动作**:
1. 追问至少以下信息:
   - 系统用了哪些云产品？（ECS/RDS/OSS 等）
   - 大约有多少台实例？
   - 系统的主要功能和用户群体
2. 如果能通过 topo-discovery 获取数据，可以跳过部分追问
3. 标注 confidence 级别

### 3.2 约束条件不明确

**症状**: 用户要求推荐架构但不告知预算、可用性要求等约束

**诊断**: 方案推荐所需的关键约束缺失

**恢复动作**:
1. 追问：
   - 预算范围
   - 目标可用性（99.9% / 99.99%）
   - 目标上线时间
   - 团队技术栈偏好
2. 如果用户无法提供，按默认值（中等预算 + 99.9% 可用性）设计
3. 在报告中标注假设条件

### 3.3 用户期望不匹配

**症状**: 用户期望本 Skill 直接操作资源（如修改安全组规则）

**诊断**: 用户对本 Skill 的能力边界有误解

**恢复动作**:
1. 明确告知本 Skill 是只读顾问，不执行任何操作
2. 对于具体操作需求，建议委托对应的产品 Skill
3. 在报告中提供清晰的 "操作建议" 步骤，说明应委托哪个 Skill 执行什么操作

## 4. 报告逻辑错误

### 4.1 架构模式分类错误

**症状**: 报告将实际是三层架构的系统错误归类为微服务

**诊断**: Generator 对拓扑数据的解读错误

**恢复动作**:
1. 检查 topo-discovery 返回的实际数据
2. 重新评估组件间的通信模式（同步 API vs 消息队列）
3. 修正模式分类并重新生成报告
4. 如果是 GCL 时发现，Critic 应给 Correctness 扣分

### 4.2 WAF 评分偏差

**症状**: 评分明显不符合实际情况（如给有严重安全问题的系统 Security 打高分）

**诊断**:
1. 检查 advisor-ops 返回的风险检查数据
2. 检查评分规则是否正确应用
3. 确认没有遗漏关键检查项

**恢复动作**:
1. 逐项复核每个支柱的检查结果
2. 修正评分并备注修正理由
3. 如果持续偏差，检查评分规则文档是否需要更新

### 4.3 推荐方案不可执行

**症状**: Mode C 推荐的方案存在技术冲突或不可行

**诊断**:
1. 检查方案中的产品组合是否兼容
2. 检查推荐规格是否在目标地域可用
3. 检查是否超出用户预算约束

**恢复动作**:
1. 重新设计方案，排除不可行选项
2. 提供至少 2 个可行方案对比
3. 标注各方案的适用场景和局限性

## 5. 环境问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `aliyun: command not found` | aliyun CLI 未安装 | 运行 `alicloud-jit-setup.sh` |
| `connect: connection refused` | 网络不通 | 检查网络代理/防火墙设置 |
| `Request expired` | 系统时间偏差 | 运行 `ntpdate` 同步时间 |
| `SignatureDoesNotMatch` | AK 不正确 | 检查 `ALIBABA_CLOUD_ACCESS_KEY_*` 环境变量 |

## 6. GCL 相关故障

| 错误 | 原因 | 恢复动作 |
|------|------|---------|
| Safety = 0 | Generator 输出了写操作建议 | **ABORT**，要求 Agent 修正，只能输出只读建议 |
| Correctness < 0.4 | 架构分析结论严重错误 | **HALT**，需要重新采集数据或追问用户 |
| Traceability < 0.6 | 报告缺少数据源标注 | 修正报告，添加数据来源和置信度标注 |
| max_iter 耗尽 | 多次迭代无法满足 GCL 标准 | 输出最后一次结果，标注 GCL 未通过 |
| 数据源降级导致评分低 | 关键数据源不可用 | 接受降级输出，明确标注局限性 |

## 7. 错误码汇总

| 错误码 | 来源 | 说明 | 恢复策略 |
|--------|------|------|---------|
| `DATA_TOPO_UNAVAILABLE` | topo-discovery | 拓扑数据不可用 | 降级为用户描述 |
| `DATA_ADVISOR_UNAVAILABLE` | advisor-ops | Advisor 数据不可用 | 跳过该数据源 |
| `DATA_CMS_EMPTY` | cms-ops | 监控指标为空 | 基于配置做定性评估 |
| `DATA_BILLING_FORBIDDEN` | billing-ops | 账单权限不足 | 跳过成本量化 |
| `INPUT_INSUFFICIENT` | 用户 | 用户信息不足以分析 | 追问 |
| `INPUT_EXPECTATION_MISMATCH` | 用户 | 用户期望本 Skill 执行操作 | 澄清边界 |
| `REPORT_CLASSIFICATION_ERROR` | 报告 | 架构模式分类错误 | 重新分类 |
| `REPORT_SCORING_BIAS` | 报告 | WAF 评分明显偏差 | 修正评分规则 |
| `REPORT_RECOMMENDATION_INFEASIBLE` | 报告 | 推荐方案不可执行 | 重新设计方案 |

## 8. 报告质量问题排查表

| 症状 | 自查步骤 | 修复方法 |
|------|---------|---------|
| 报告内容空泛 | 是否完成了数据采集？是否追问了用户？ | 补充数据或追问用户 |
| 评分全部偏高 | 是否忽略了风险检查结果？ | 降低评分至合理范围 |
| 缺少具体建议 | 是否完成了改进建议阶段？ | 针对每个发现给出建议 |
| 方案对比不足 | Mode C 是否只提供了一个方案？ | 补充第二个可行方案 |
| 缺乏数据源说明 | 是否记录了数据来源？ | 添加 data_sources 记录 |
| 用户看不懂 | 是否使用了过多技术术语？ | 用业务语言重新表述 |
