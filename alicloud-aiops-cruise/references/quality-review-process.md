# 质量门定期评审流程

> **每个 Sprint 的交付质量不是一次性的——它需要持续的审视和优化。**
> 本文件定义质量门的定期评审机制，确保质量标准随系统演进不断提升。

---

## 一、三层评审周期

```
频率       范围                     触发方式                   负责人
────────────────────────────────────────────────────────────────────
每 Sprint   当前 Sprint 的质量门     标记 Sprint 完成前          Agent / 开发者
每周        所有活跃 Sprint 的质量门  周一自动检查                Agent
每月        全部质量门的趋势分析      月底自动生成报告             Agent
```

### 1.1 Sprint 级评审（门禁）

**触发时机**：每个 Sprint 标记 `[x]` 前

**动作**：

```bash
# 运行当前 Sprint 的所有质量门
for q in $(grep -oP 'Q\d+\.\d+' TODO.md | sort -u); do
  echo "Running $q..."
  # 执行对应的检查命令
done
# 全部通过 → 标记 Sprint 完成
# 有失败 → 修复后重新检查
```

**输出**：质量门报告（Markdown，追加到 Sprint 完成记录）

### 1.2 每周评审（健康检查）

**触发时机**：每周一自动执行

**动作**：

| 检查项 | 方法 | 异常处理 |
|--------|------|---------|
| 所有质量门仍可通过 | 重新执行所有 Q 检查 | 失败项标记为回归，安排修复 |
| 是否有误报/漏报 | 抽查最近 3 次巡检报告的人工反馈 | 统计误报率，超过阈值则调优 |
| 阈值是否需要调整 | 对比真实告警和系统告警 | 调优 Z-Score 阈值或降噪策略 |
| 新增指标是否覆盖 | 检查是否有新指标没纳入基线 | 更新指标映射表 |

**输出**：`audit-results/reviews/weekly-{date}.md`

### 1.3 月度评审（趋势分析）

**触发时机**：每月 1 号

**动作**：

1. 统计本月各质量门的通过率趋势
2. 统计误报率（FP）和漏报率（FN）的变化
3. 分析是否有需要新增/删除/调整的质量门
4. 生成趋势图表（文本表格即可）

**输出**：`audit-results/reviews/monthly-{month}.md`

## 二、质量门生命周期

```
创建 → 验证 → 监控 → 评估 → 优化/淘汰
 │                         ↑
 └─────────────────────────┘
        持续循环
```

### 2.1 创建

新增质量门的条件：

- 该 Sprint 有新的交付物类型
- 历史数据表明某项检查频繁失败
- 发现质量漏洞需要新的防护

### 2.2 验证

新质量门加入后，需连续 3 个 Sprint 通过才能视为稳定。

### 2.3 监控

每月评审时检查每个质量门的：

| 指标 | 含义 | 异常值 |
|------|------|--------|
| 通过率 | 该门检查通过次数 / 总检查次数 | < 95% 说明门太严或代码质量差 |
| 误报率 | 门检查失败但实际没问题 | > 5% 说明门需要调整 |
| 执行耗时 | 该门检查的平均耗时 | > 30s 说明检查太慢，需要优化 |
| 覆盖率 | 该门覆盖的交付物范围 | < 80% 说明有盲区 |

### 2.4 优化

| 发现 | 动作 |
|------|------|
| 通过率 < 80% 且误报率 > 10% | 调整阈值或检查逻辑 |
| 执行耗时 > 30s | 优化检查脚本或合并检查 |
| 连续 3 个月通过率 = 100% | 降级为"信用检查"（抽查模式） |
| 覆盖率 < 80% | 新增质量门补盲区 |

### 2.5 淘汰

质量门连续 6 个月通过率 = 100% 且误报率 = 0% → 可以考虑移除（或转为低优先级）。

## 三、质量报告格式

### Sprint 级报告

```markdown
## Sprint {n} 质量门报告

| 门编号 | 检查项 | 结果 | 耗时 |
|--------|--------|------|------|
| Q1.1 | Shell 语法 | ✅ PASS | 0.3s |
| Q1.2 | 参数解析 | ✅ PASS | 0.1s |
| Q1.3 | 凭证泄露 | ✅ PASS | 0.2s |

结论: ✅ 全部通过 (3/3)
```

### 月度趋势报告

```markdown
## 质量趋势报告 - 2026年6月

### 通过率趋势
| 周次 | 总门数 | 通过 | 通过率 |
|------|--------|------|--------|
| W1   | 12     | 12   | 100%   |
| W2   | 12     | 11   | 91.7%  |
| W3   | 14     | 14   | 100%   |
| W4   | 14     | 13   | 92.9%  |

### 异常检测精度
| 指标 | 本月 | 上月 | 变化 |
|------|------|------|------|
| 误报率 (FP) | 3.2% | 4.1% | 📉 改善 |
| 漏报率 (FN) | 1.8% | 2.5% | 📉 改善 |
| 平均 Z-Score 阈值 | 2.5 | 2.5 | ➡️ 持平 |

### 建议
1. 本月 W2 的 Q2.2 因网络抖动失败 → 增加重试逻辑
2. SLS 接入后需新增 2 个质量门
```
