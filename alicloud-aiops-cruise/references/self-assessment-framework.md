# AIOps Cruise 能力自评估框架

> 定义三阶段演进模型及各阶段准入/验收标准。
> 每次迭代结束后，对照本文件做自评估，明确当前所处阶段。

---

## 一、评估维度总览

| # | 维度 | 说明 | 权重 |
|---|------|------|------|
| D1 | **执行模式** | Agent 如何执行巡检 | 25% |
| D2 | **检测精度** | 如何发现异常 | 20% |
| D3 | **自动化深度** | 发现问题后如何处理 | 20% |
| D4 | **数据广度** | 覆盖哪些数据源 | 15% |
| D5 | **学习能力** | 是否从历史中改进 | 10% |
| D6 | **交付完整性** | 输出产物是否达标 | 10% |

## 二、三阶段定义

### Stage 1: 规则引擎巡检（当前 v1.0）

| 维度 | 状态 | 判定标准 |
|------|------|---------|
| D1 执行 | Agent 读取 MD -> LLM 推理 -> 生成命令 | 执行路径经过 LLM |
| D2 检测 | 固定阈值 (CPU>70%=Warning) | 无动态基线 |
| D3 自动化 | 纯读，所有写操作建议需人工确认 | 无白名单、无自动执行 |
| D4 数据 | CloudMonitor + DAS + ActionTrail | 无 SLS/ARMS |
| D5 学习 | 无 Incident 记录 | 无历史复用 |
| D6 交付 | Markdown + JSON | 无评分看板 |

**进入标准**: 无（初始状态）

**验收标准（满足以下全部可进入 Stage 2）**:

- [ ] D1: Runbook 脚本化完成，Agent 可直接 bash 调用
- [ ] D2: 动态基线 Z-Score 嵌入 01 号 runbook
- [ ] D3: 预授权白名单文档初版完成（≥3 项）
- [ ] D4: 拓扑渲染联动完成（topo-render.py --health-json）
- [ ] D5: Incident Schema 定义完成
- [ ] D6: 三份交付物标准定义完成（本文件）

---

### Stage 2: 半自动智能巡检（v1.5）

| 维度 | 状态 | 判定标准 |
|------|------|---------|
| D1 执行 | Agent 调 Bash 脚本直接执行 | 执行路径不经过 LLM 推理 |
| D2 检测 | 动态基线 Z-Score + 固定阈值双重判定 | 两种方法取最高等级 |
| D3 自动化 | 低风险白名单操作可自动执行 | 自动执行需记录 trace |
| D4 数据 | CloudMonitor + DAS + SLS 日志 | 可查应用层 error 日志 |
| D5 学习 | Incident DB 记录 + 历史检索 | 排查时可查相似案例 |
| D6 交付 | Markdown + 评分看板 + JSON | 三份齐全 |

**进入标准（满足以下全部）**:

- [ ] Stage 1 全部验收项完成
- [ ] 至少完成 3 次真实环境巡检，验证流程正常

**验收标准（满足以下全部可进入 Stage 3）**:

- [ ] D1: 4 个 runbook 全部脚本化 + 幂等加固完成
- [ ] D2: Python STL 时序分解引擎 `baseline-scorer.py` 运行稳定
- [ ] D3: 白名单操作自动执行 ≥ 50 次无安全事故
- [ ] D4: SLS 日志集成 + 至少 2 条 SLS 推理规则
- [ ] D5: Incident DB ≥ 10 条记录
- [ ] D6: 三份交付物自动生成，无需人工干预

### Stage 3: 双引擎自治巡检（v2.0）

| 维度 | 状态 | 判定标准 |
|------|------|---------|
| D1 执行 | 固化工作流自触发 + 弹性 Agent 兜底 | 调度器自动路由 |
| D2 检测 | STL 时序分解 + 分位数 + Z-Score 混合 | 按指标自动选择最优方法 |
| D3 自动化 | 条件全自动（1000+ 次成功操作自动解锁） | 人工确认仅用于高风险 |
| D4 数据 | CloudMonitor + DAS + SLS + ARMS | 四层全链路可观测 |
| D5 学习 | Incident DB + Post-Mortem 自动复盘 | 相似故障秒级匹配 |
| D6 交付 | 三份交付物 + 推送到钉钉/企微/Grafana | 自动分发 |

**进入标准（满足以下全部）**:

- [ ] Stage 2 全部验收项完成
- [ ] workflow-runner.py 稳定运行 ≥ 30 天
- [ ] D0 待讨论决策全部完成并实现

**验收标准（v2.0 最终目标）**:

- [ ] D1: 固化工作流延迟 < 1s；弹性 Agent 覆盖率 > 90%
- [ ] D2: 误报率 < 5%，漏报率 < 10%
- [ ] D3: 白名单操作成功率 > 99%，0 安全事故
- [ ] D4: 四层数据源全部在线
- [ ] D5: Incident DB ≥ 100 条可检索记录
- [ ] D6: 交付物自动分发，管理层可每日查看评分看板

## 三、自评估操作流程

### 每次迭代结束时的自检

```bash
# 运行 Sprint 状态真值表脚本（P0-3）
python3 scripts/sprint-status-truth-table.py

# 校验 TODO.md 索引与 Sprint 文件一致性
python3 scripts/sprint-status-truth-table.py --verify

# 查看指定 Sprint 的 JSON 详情
python3 scripts/sprint-status-truth-table.py --sprint 20 --json

# 输出示例:
# # Sprint 状态真值表
# | Sprint | 名称 | P | 任务 | 质量门 | 自检 | 文件状态 | 总 verdict |
# |--------|------|---|------|--------|------|----------|------------|
# | 1 | 核心脚本化 | P0 | 11/11 ✅ | N/A | N/A | — ✅ | ✅ PASS |
# ...
# **汇总**: 9/17 Sprint 全部通过 | 通过率 52.9%
```

### 人工自检流程

1. 打开本文件
2. 逐项对照当前阶段的"验收标准"清单
3. 已完成的打 PASS，未完成的打 [ ]
4. 统计完成率 = (已完成项 / 总项) × 100%
5. 根据完成率决定是否进入下一阶段：
   - < 50%: 继续当前阶段
   - 50-99%: 可以并行启动下一阶段的准备工作
   - 100%: 正式进入下一阶段
