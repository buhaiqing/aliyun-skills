# Well-Architected Assessment — 五支柱评估

---

## Security (安全)

| 原则 | 实施方式 |
|------|---------|
| **最小权限委托** | auto-scaling-orch 不直接持有 AK，通过 Agent 运行时环境变量传递。下游 Skill 使用各自的最小权限 RAM 策略 |
| **凭证隔离** | 严格遵守 `{{env.*}}` ONLY 规范，NEVER 在日志/输出中暴露 AK |
| **操作安全门** | 三级安全门: 🟢 自动 / 🟡 白名单 / 🔴 需确认 + ⛔ 禁止熔断 |
| **审计追踪** | 每次编排操作记录完整链路 (TraceID, 请求/响应, 耗时, 状态) |
| **变更控制** | 🔴 级操作需用户明确确认，确认记录写入审计日志 |

## Reliability (可靠性)

| 原则 | 实施方式 |
|------|---------|
| **5 步标准流程** | 感知→决策→编排→执行→验证，缺一不可 |
| **防抖动机制** | 冷却时间 + 方向反转检测 + 小波动过滤 + 连续触发递减 |
| **熔断保护** | 5 种熔断条件：频繁触发、活动未完成、余额不足、配额不足、方向反转 |
| **回滚策略** | 每个场景定义明确回滚动作，验证失败自动触发 |
| **幂等重试** | 每个写操作使用 ClientToken (UUID v4)，支持幂等重试 |

## Cost (成本)

| 原则 | 实施方式 |
|------|---------|
| **闲置回收** | S6 场景直接节省：自动识别低负载资源并缩至 MinSize |
| **定时缩容** | S2 场景：非工作时间缩容，避免资源闲置浪费 |
| **预测性扩缩** | S3 场景：基于历史规律精准扩缩，避免过度预留 |
| **成本对比** | 每次扩缩操作报告预估成本影响（扩前/扩后对比） |
| **RI/SP 推荐** | 发现弹性实例长期运行 → 建议 RI/SP 节省更多 |

## Efficiency (效率)

| 原则 | 实施方式 |
|------|---------|
| **自动化编排** | 6 场景覆盖 90%+ 弹性需求，Agent 自动决策 + 编排 |
| **5 步法标准化** | 统一的编排流程，新增场景只需定义 5 步 |
| **并行采集** | 多指标采集使用并行调用 (S4)，决策 < 5s |
| **渐进式深度模式** | S3 先做周期性检测，无周期自动降级到 S1 |
| **策略模板复用** | 预置策略模板 (assets/example-policies/)，快速部署 |

## Performance (性能)

| 原则 | 目标 | 说明 |
|------|:----:|------|
| **指标采集** | < 5s | 委托 cms-ops DescribeMetricLast 单次 < 1s |
| **决策计算** | < 1s | 本地规则匹配 + 容量公式计算 |
| **编排指令** | < 10s | 委托 ess-ops CreateScalingRule 平均 2-3s |
| **完整流程** | < 60s | 不含 ESS 实例创建/启动等待时间 |
| **实例预热** | 120-300s | EstimatedInstanceWarmup 参数控制 |
| **冷却时间** | 300-600s | 防止频繁触发 |

---

## 评估总结

| 支柱 | 评分 | 评价 |
|:----:|:----:|------|
| Security | ★★★★☆ | 安全门 + 熔断完善，但依赖下游 Skill 的 RAM 策略是否正确配置 |
| Reliability | ★★★★★ | 5 步法 + 熔断 + 回滚 + 幂等，闭环完整 |
| Cost | ★★★★★ | 3 个直接省钱场景 + 成本报告，效率最高 |
| Efficiency | ★★★★★ | 6 场景标准化 + 策略模板复用，运维效率显著提升 |
| Performance | ★★★★☆ | 决策快，但实例启动时间受云平台限制无法优化 |
